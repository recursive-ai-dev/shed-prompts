<audit_scope>
Search strictly for the following categories:
- Hardcoded Strings: User-facing text embedded directly in code/markup instead of routed through the translation layer.
- Locale-Sensitive Formatting: Hardcoded date, time, number, or currency formatting that doesn't account for locale conventions (decimal separators, date order, currency symbol placement).
- Pluralization: String concatenation or manual logic handling plural forms instead of using the i18n framework's pluralization rules (which vary far more than singular/plural across languages).
- Layout Assumptions: Fixed-width containers or truncation logic that will break for languages with longer average string length or right-to-left (RTL) scripts.
- RTL Readiness: Hardcoded `left`/`right` CSS properties or directional icons/logic that won't flip correctly for RTL locales.
- Concatenation Fragility: Strings built by concatenating translated fragments, which breaks grammar and word order in many target languages.
</audit_scope>

<negative_constraints>
- DO NOT introduce a new i18n library if the project already has one — extend the existing pattern.
- DO NOT machine-translate actual UI copy into other languages as part of this pass — extraction and scaffolding only, unless the user explicitly provides translated strings or requests translation.
- DO NOT flatten pluralization into a single generic string — use the framework's plural-rule support even if only English is currently supported, so the scaffolding is correct when new locales are added.
- DO NOT hardcode locale-specific formatting logic manually when the runtime/framework already provides locale-aware formatting utilities (e.g., `Intl` in JS).
- DO NOT pad the list to hit a specific count.
</negative_constraints>

<implementation_standards>
- Every extracted string must use a stable, human-readable translation key (not an auto-generated hash) so translators have context.
- String concatenation for sentences must be replaced with parameterized/templated translation strings that allow full sentence reordering per locale.
- All date/time/number/currency output must route through locale-aware formatting utilities, defaulting to the project's current locale as the fallback.
</implementation_standards>
